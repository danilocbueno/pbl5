\section{Descrição da aplicação analisada e do problema investigado}
% =============================================================================

Este trabalho propõe uma estratégia de avaliação para a qualidade de dois modelos de linguagem de larga escala (LLMs) especializados em geração de código: \texttt{qwen2.5-coder:1.5b} e \texttt{deepseek-coder:1.3b}. Ambos os modelos foram executados localmente em um Mac Mini M4 por meio da plataforma Ollama\footnote{\url{https://ollama.com}}, sem acesso a recursos de nuvem ou API externa. Os modelos foram selecionados pelo seu formato específico em geração de código e sua popularidade.

O problema investigado consiste em avaliar em que medida esses modelos são capazes de gerar código Python funcionalmente correto para tarefas simples, e se essa capacidade se mantém consistente quando o mesmo problema é apresentado sob diferentes formulações de \textit{prompt}.

A escolha da geração de código como domínio de avaliação justifica-se pela sua capacidade de mitigar o \textit{Oracle Problem}, amplamente discutido na literatura \cite{barr2015oracle}: diferentemente de tarefas de geração de texto livre, a corretude de um trecho de código pode ser verificada objetivamente por meio da execução de testes unitários. Isso permite que a avaliação seja automatizada e reproduzível.

Os cinco problemas escolhidos cobrem funcionalidades simples em Python:

\begin{itemize}
    \item \textbf{P1} — encontrar o maior número de uma lista;
    \item \textbf{P2} — verificar se uma string é palíndromo, ignorando maiúsculas e espaços;
    \item \textbf{P3} — calcular o fatorial de um número inteiro de forma recursiva;
    \item \textbf{P4} — contar vogais em uma string, sem distinção de maiúsculas;
    \item \textbf{P5} — retornar os dois maiores valores de uma lista como uma tupla ordenada.
\end{itemize}

\section{Fundamentação da estratégia de avaliação adotada}
% =============================================================================
A estratégia adotada organiza-se em torno de três decisões metodológicas
interdependentes.

A primeira é a resolução do \textit{Oracle Problem} \cite{barr2015oracle}
pela escolha do domínio de avaliação. Para tarefas de geração de texto livre,
não existe critério objetivo e automatizável para julgar a qualidade de uma
resposta. A geração de \textbf{código Python} resolve esse problema: a
corretude funcional de um trecho de código é verificável objetivamente pela
execução de testes unitários. Para cada um dos cinco problemas, o grupo
elaborou previamente uma bateria de 7 a 10 testes unitários cobrindo casos
gerais, extremos e negativos, escritos antes da coleta das respostas para
evitar viés de confirmação.

A segunda é a avaliação de \textbf{robustez de corretude} sob variações
controladas de \textit{prompt}. Essa abordagem é motivada por dois trabalhos
da literatura. Zhu et al.\ \cite{zhu2023promptrobust} demonstraram que LLMs
são sensíveis a variações semânticas neutras nos prompts de instrução, e que
a avaliação com um único \textit{prompt} fixo subestima a fragilidade real
do modelo. Chan et al.\ \cite{chan2025symmetric} partiram da mesma intuição
aplicada a LLMs de geração de código: se dois prompts descrevem o mesmo
problema, os modelos deveriam se comportar de forma equivalente — e violações
dessa expectativa revelam instabilidade. Este trabalho combina essas duas
perspectivas, usando variações de prompt para medir não consistência de saída,
mas robustez de corretude verificada por testes unitários. Seguindo esse
princípio, cada problema é apresentado sob seis formulações distintas, e a
variação na taxa de aprovação entre formulações é usada como métrica de
instabilidade:


\begin{itemize}
    \item \textbf{Enunciado direto} (\texttt{original}): formulação de
    referência em português;
    \item \textbf{Reformulação semântica} (\texttt{reformulado}): reescrita
    com vocabulário diferente, mantendo o significado e os requisitos
    funcionais;
    \item \textbf{Mudança de idioma} (\texttt{ingles}): tradução do enunciado
    para o inglês, preservando a especificação;
    \item \textbf{Detalhamento excessivo} (\texttt{verboso}): enunciado longo
    com informações redundantes e explicitação de casos triviais já implícitos
    no original;
    \item \textbf{Simplificação} (\texttt{vago}): enunciado mínimo, sem
    especificações além do essencial;
    \item \textbf{Inserção de ruído semântico} (\texttt{com\_ruido}): adição
    de informação contextual irrelevante para a solução antes do enunciado
    real.
\end{itemize}

É importante notar que essa métrica mede robustez de \textit{corretude} ---
quanto a capacidade de resolver corretamente o problema varia com a formulação
---, não consistência de saída entre variantes. Um modelo que gera
consistentemente o mesmo código incorreto para todas as variantes registra
variação zero sem ter resolvido nenhuma instância do problema; por isso os
testes unitários, e não a comparação entre outputs, são o critério central.

A terceira é a separação entre \textbf{falhas estruturais e estocásticas}.
LLMs com temperatura positiva são não determinísticos: a mesma combinação
modelo~$\times$~\textit{prompt} pode produzir saídas distintas em execuções
diferentes, tornando difícil distinguir limitações genuínas do modelo de
resultados desfavoráveis por acaso. Para separar esses dois fenômenos, os
experimentos foram conduzidos em dois regimes: temperatura padrão, executado
cinco vezes por combinação para quantificar a variância estocástica; e
temperatura zero, que desabilita a amostragem e torna os resultados
determinísticos. Falhas que persistem em ambos os regimes são classificadas
como estruturais; falhas que aparecem apenas em um dos regimes são
classificadas como estocásticas.

\subsection{Coleta de Respostas}

Os modelos foram instruídos por meio de um \textit{system prompt}
explícito a responderem apenas com código Python puro, sem explicações
ou formatação adicional:

\begin{lstlisting}[style=python, caption={System prompt utilizado na
coleta de respostas}]
Você é um assistente de programação Python. Responda APENAS com o código Python solicitado, sem explicações, sem markdown, sem blocos de código cercados por ```. Apenas o código puro.
\end{lstlisting}

As respostas foram coletadas via API local do Ollama. Cada uma das 60 combinações modelo~$\times$~variante foi executada uma vez e a resposta bruta salva em \texttt{generated\_code.json}.

Apesar da instrução de formato, o \texttt{deepseek-coder} frequentemente ignorou o \textbf{}{system prompt} e incluiu explicações junto ao código. Esse comportamento motivou a etapa de extração descrita a seguir.

\subsection{Extração de Código}

Antes da avaliação, um script processou as respostas brutas, separando o código Python dos textos de explicação. A
extração utilizou as seguintes heurísticas:

\begin{enumerate}
    \item Remoção de blocos markdown (\texttt{```python ... ```});
    \item Validação do texto completo via \texttt{ast.parse} — se já
    era código válido, retornado diretamente;
    \item Filtragem linha a linha, descartando linhas de prosa sem
    prefixo de comentário (\texttt{\#});
    \item Remoção de chamadas a \texttt{print()} e \texttt{input()}
    no nível do módulo;
    \item Normalização de indentação (conversão de tabs para 4 espaços).
\end{enumerate}

Para garantir uma comparação justa — avaliando a capacidade de
resolução do problema e não a aderência ao formato —, apenas os casos
com código sintaticamente válido após a extração foram avaliados; os
demais foram contabilizados como falha de geração para aquela execução.
A quantidade de casos inválidos é, ela própria, \textit{estocástica}
sob temperatura padrão: variou entre 9 e 14 dos 60 casos por execução
(11 na primeira run), em sua maioria — mas nem sempre exclusivamente —
do \texttt{deepseek-coder} (nas runs 2 e 3 o \texttt{qwen2.5-coder}
também gerou código inválido). Sob temperatura zero (determinística),
esse número cai para 5 casos, todos do \texttt{deepseek-coder}. Apenas
estes últimos podem ser considerados falhas estruturais de geração; os
demais são, em princípio, recuperáveis por reamostragem.

Os dois exemplos a seguir ilustram casos em que o próprio modelo
constrói uma expressão Python incorreta — não prosa remanescente nem
deslizes de indentação, mas confusão genuína sobre a sintaxe da
construção que estava tentando escrever.

O primeiro, do \texttt{deepseek-coder} para \texttt{P3\_original}
(\textit{prompt}: ``Escreva uma função Python chamada \texttt{fatorial}
que calcula o fatorial de um número inteiro \texttt{n} usando
recursão\dots''), executado com temperatura padrão:

\begin{lstlisting}[style=python, caption={deepseek-coder em P3\_original — operador ternário malformado}]
def fatorial(n):
    if not isinstance(n, int) or n < 0:
        raise ValueError('O argumento deve ser um valor positive integer')

    return 1 if (n == 0 else n * fatorial(n-1))
\end{lstlisting}

A última linha tenta escrever um operador ternário (\texttt{X if
condição else Y}), mas o modelo o constrói de forma equivocada,
envolvendo a condição em parênteses e posicionando o \texttt{else}
\textit{dentro} deles — uma confusão entre a sintaxe do ternário Python
e a de outras linguagens (como o \texttt{cond ? a : b} de C/Java). O
\texttt{ast.parse} rejeita com \textit{invalid syntax} na própria linha
do erro: não há prosa a remover nem indentação a corrigir — a
construção em si é sintaticamente impossível.

O segundo, do \texttt{deepseek-coder} para \texttt{P5\_vago}
(\textit{prompt}: ``Função Python \texttt{dois\_maiores} que retorna os
dois maiores de uma lista como tupla''), também sob temperatura padrão:

\begin{lstlisting}[style=python, caption={deepseek-coder em P5\_vago — aninhamento de parênteses e colchetes fora de controle}]
def dois_maiores(lst):
    lst = list(set([i for i in set(lst) if type(i)!=str]))
    lst = [x for x in set([int(y) if isinstance(y, str) and y[0] != '-'
           else float('-inf')]) | sort(), 2)]))]))))))
    return tuple(lst[-1:]) + lst[0] if len(lst) >= 2 else None
\end{lstlisting}

Ao tentar compor uma expressão de \textit{list comprehension} aninhada
com conversões de tipo, o modelo perde o controle da própria construção:
a segunda linha acumula chamadas e colchetes que não fecham na ordem em
que abrem, terminando em uma sequência de seis parênteses/colchetes de
fechamento sem abertura correspondente. O \texttt{ast.parse} rejeita com
\textit{closing parenthesis `)' does not match opening parenthesis
`['}. Em ambos os exemplos, o erro está na lógica estrutural que o
modelo tentou expressar — não há separação possível entre ``código'' e
``prosa'' a ser corrigida pela extração; apenas a reamostragem poderia,
eventualmente, produzir uma construção sintaticamente válida.

\subsection{Avaliação}

O script \texttt{rodar\_testes.py} executou cada código extraído
contra a bateria de testes unitários via \texttt{exec()} em
\textit{namespace} isolado. Para cada caso, foi verificado se o
resultado obtido era igual ao esperado. A taxa de aprovação foi
calculada por combinação modelo~$\times$~variante, e a variação entre
variantes do mesmo problema foi usada como métrica de robustez de
corretude.

\section{Apresentação e análise dos Resultados}
% =============================================================================

Foram realizados dois experimentos com as mesmas 60 combinações modelo~$\times$~variante. O primeiro, com \textbf{temperatura zero} (determinístico), força o modelo a escolher sempre o token de maior probabilidade: o mesmo \textit{prompt} produz exatamente o mesmo código a cada execução, estabelecendo uma base reproduzível do comportamento de cada formulação. O segundo, com \textbf{temperatura padrão} do Ollama (estocástico, que representa o uso típico), foi repetido cinco vezes para caracterizar a variância introduzida pela amostragem.

A análise é apresentada do comportamento mais reproduzível ao mais variável: primeiro a temperatura zero, em seguida a dispersão das cinco execuções com temperatura padrão e, por fim, a síntese que distingue falhas \textit{estruturais} — presentes independentemente da aleatoriedade — de falhas \textit{estocásticas}, que dependem do acaso na amostragem do modelo.

\subsection{Temperatura Zero — Comportamento Determinístico}

\textbf{qwen2.5-coder.} P1, P2, P3 e P4 atingem 0pp de variação — consistência perfeita em 24 das 30 combinações. P5 é a única instabilidade: três variantes geram código correto (reformulado, inglês e com ruído) e as outras três falham, cada uma com um bug distinto:

\begin{lstlisting}[style=python, caption={Falhas de qwen2.5-coder em P5 com temperatura zero}]
# P5_original — tupla correta, mas ordem crescente (14% = 1/7)
def dois_maiores(nums):
    return tuple(sorted(nums)[-2:])

# P5_verboso — ordem correta, mas retorna lista (0%)
def dois_maiores(nums):
    nums.sort(reverse=True)
    return nums[:2]

# P5_vago — usa set() e ordem crescente (0%)
def dois_maiores(lista):
    return tuple(sorted(set(lista))[-2:])
\end{lstlisting}

\textbf{deepseek-coder.} O comportamento é muito mais sensível à formulação do \textit{prompt}, sem um estilo de enunciado consistentemente eficaz:

\begin{itemize}
    \item \textbf{P3:} \texttt{P3\_original}, \texttt{P3\_reformulado} e \texttt{P3\_vago} atingem 100\%; \texttt{P3\_ingles} cai para 88\% por um bug sutil — o caso base retorna \texttt{n} em vez de \texttt{1}, fazendo \texttt{fatorial(0) = 0}; \texttt{P3\_verboso} e \texttt{P3\_com\_ruido} ficam em 0\%.
    \item \textbf{P4:} \texttt{P4\_original}, \texttt{P4\_ingles}, \texttt{P4\_verboso} e \texttt{P4\_com\_ruido} atingem 100\%; \texttt{P4\_vago} fica em 0\% (código malformado). O destaque negativo é \texttt{P4\_reformulado}, que gera uma função \texttt{count\_consonants} contando \textit{consoantes} em vez de vogais — um caso extremo de ``problema trocado'', com a lógica completamente invertida.
    \item \textbf{P2:} nunca atinge 100\%. \texttt{P2\_ingles} e \texttt{P2\_verboso} alcançam apenas 70\% pois ambos geram \texttt{return s == s[::-1]}, que ignora a normalização de maiúsculas e espaços; as demais variantes ficam em 0\%.
    \item \textbf{P1:} apenas \texttt{P1\_verboso} falha (0\%); as demais cinco variantes atingem 100\%.
\end{itemize}

Assim como ocorre com o \texttt{qwen2.5-coder}, P5 é onde o \texttt{deepseek-coder} concentra a maior diversidade de bugs: três variantes produzem código parcialmente correto (\texttt{original}, 71\%; \texttt{inglês} e \texttt{verboso}, 86\%) e três falham por completo (0\%), cada uma de um modo distinto:

    numeros = list(set(numeros))
    ...

# P5_reformulado — retorna tupla aninhada com elementos extras, não (maior, segundo_maior) (0%)
def dois_maiores(numeros):
    numeros.sort()
    return (numeros[-1], numeros[-2]) if len(numeros) > 1 else None, max(numeros), min(numeros)

# P5_vago — tipo de retorno inconsistente: string no caso geral, tupla só para listas de tamanho 3 (0%)

    ...
    return (maior1, maior2) if len(lista) == 3 else "Lista com menos que dois elementos"
\end{lstlisting}

\textbf{qwen2.5-coder.} A maioria das combinações de P1–P4 apresenta $\sigma = 0$ ou baixo, indicando comportamento estável. Dois bugs ilustram falhas que aparecem apenas em algumas runs e que, sob temperatura zero, não ocorrem. Em \texttt{P2\_com\_ruido} (média 94\,±\,13), uma das execuções gerou normalização incompleta:

\begin{lstlisting}[style=python, caption={qwen2.5-coder em P2\_com\_ruido (uma das runs) — normalização de maiúsculas incompleta}]
def palindromo(s):
    return s.lower() == s[::-1]
\end{lstlisting}

O \texttt{s.lower()} normaliza a string, mas \texttt{s[::-1]} a inverte \textit{sem} normalizar, e a comparação falha para palíndromos com maiúsculas (\texttt{"Arara"}, \texttt{"Ana"}). Em \texttt{P4\_ingles} (média 86\,±\,31), outra execução usou \texttt{str.count} de forma incorreta:

\begin{lstlisting}[style=python, caption={qwen2.5-coder em P4\_ingles (uma das runs) — erro de semântica da API}]
def contar_vogais(texto):
    return texto.lower().count('aeiou')
\end{lstlisting}

A chamada \texttt{count('aeiou')} procura a \textit{substring} inteira em vez de cada vogal individualmente.

P5 é o problema mais instável do qwen: mesmo as variantes de melhor média (\texttt{P5\_reformulado}, 71\,±\,40, e \texttt{P5\_original}, 60\,±\,55) erram em parte considerável das execuções, e os desvios elevados (frequentemente próximos de 50pp) indicam execuções que oscilam entre acerto e falha total. Apenas \texttt{P5\_verboso} é consistentemente 0\% (0\,±\,0), uma falha estrutural. Os bugs típicos envolvem tipo de retorno e ordenação — por exemplo, retornar uma lista em ordem crescente em vez de uma tupla decrescente:

\begin{lstlisting}[style=python, caption={qwen2.5-coder em P5\_ingles — erro de tipo de retorno}]
def dois_maiores(numbers):
    return sorted(numbers)[-2:]
\end{lstlisting}

ou eliminar duplicatas com \texttt{set()}, falhando no caso \texttt{[5,5,5]} que deveria retornar \texttt{(5,5)}:

\begin{lstlisting}[style=python, caption={qwen2.5-coder em P5\_verboso — retorna lista com set (perde duplicatas)}]
def dois_maiores(nums):
    sorted_nums = sorted(set(nums))
    return sorted_nums[-2:]
\end{lstlisting}

\textbf{deepseek-coder.} A Tabela~\ref{tab:multi_runs} confirma alta variância generalizada. As únicas combinações estáveis são as que ficam consistentemente em 0\% (P2\_original, P2\_reformulado, P2\_vago, P2\_com\_ruido, P5\_vago, P5\_com\_ruido) ou em 100\% (P1\_original, P1\_reformulado, P1\_ingles, P1\_vago, P1\_com\_ruido) — indicando falhas e sucessos estruturais, respectivamente. Tudo mais é essencialmente aleatório, com desvios padrão frequentemente superiores a 40pp.

Um padrão recorrente ajuda a explicar boa parte dessa instabilidade — e em especial os 0\% quase universais de P2 e P5: o \texttt{deepseek-coder} frequentemente \textbf{ignora o nome de função especificado no \textit{prompt}}, mesmo quando o resto da implementação está correto. Em P2 (nome esperado: \texttt{palindromo}), das 30 gerações sintaticamente válidas ao longo das cinco runs, apenas 5 usaram o nome pedido; as demais geraram \texttt{palindrome} (13 vezes), \texttt{is\_palindrome} (5) ou até \texttt{is\_anagram} (2) — todas reprovadas em 100\% dos casos de teste, independentemente da corretude do algoritmo:

\begin{lstlisting}[style=python, caption={deepseek-coder em P2\_original (run 3) — lógica correta, nome de função errado (0\%)}]
def palindrome(s):                       # esperado: palindromo
    s = ''.join(filter(str.isalnum, s)).lower()
    return s == s[::-1]
\end{lstlisting}

O mesmo padrão se repete em P5 (nome esperado: \texttt{dois\_maiores}): de 24 gerações válidas, apenas 11 usaram o nome correto, e as demais variaram entre \texttt{dos\_maiores} (5), \texttt{duas\_maiores} (2) e até \texttt{duos\_maiores} (1) — pequenos erros de digitação que, isoladamente, já bastam para zerar a taxa de aprovação:

\begin{lstlisting}[style=python, caption={deepseek-coder em P5\_vago (run 5) e P5\_com\_ruido (run 1) — variações no nome de "dois\_maiores" (0\% em ambos)}]
def duos_maiores(numeros):               # run 5 — esperado: dois_maiores
    ...

def duas_maiores(lst):                   # run 1 — esperado: dois_maiores
    ...
\end{lstlisting}

Esse comportamento já havia surgido isoladamente em temperatura zero (\texttt{P5\_com\_ruido} também gerou \texttt{dos\_maiores} ali), mas só a repetição ao longo de 30 amostras por problema revela sua extensão: para o \texttt{deepseek-coder}, a falha em aderir ao \textbf{contrato de interface} (nome exato da função) é, em si, uma fonte de erro tão relevante quanto bugs de lógica — e infla artificialmente a aparência de instabilidade em P2 e P5, quando na verdade boa parte do código gerado resolveria o problema corretamente sob outro nome.

Um segundo achado, de natureza distinta, emergiu na quarta run para o caso \textbf{deepseek-coder~|~P3\_reformulado}. O modelo gerou o seguinte código:

\begin{lstlisting}[style=python, caption={deepseek-coder em P3\_reformulado (run 4) — multiplicação de string exponencial}]
def fatorial(n):
    if n == 0 or (n < 0 and not isinstance(n, int)):
        return 'Por favor informe número positivo'
    elif not isinstance(n, int):
        return 'Por favor informe número positivo'
    elif n == 1:
        return 'Por favor informe outros valores'
    else:
        return n * fatorial(n - 1)
\end{lstlisting}

O código é sintaticamente válido e estruturalmente recursivo, mas os casos base retornam \textbf{strings de mensagem de erro} em vez de inteiros. Como consequência, a linha \texttt{return n * fatorial(n - 1)} executa \textit{multiplicação de string} em Python (\texttt{n * str} repete a string \texttt{n} vezes), não multiplicação numérica. O resultado de \texttt{fatorial(10)} é a string repetida $10! = 3.628.800$ vezes — cerca de 116 milhões de caracteres, que inflaram o arquivo de resultados dessa run para 116~MB. Nas outras quatro runs o mesmo \textit{prompt} gerou código diferente e nenhuma reproduziu o bug: com execução única o fenômeno seria invisível. O caso reforça também a necessidade de \textit{sandboxing} robusto (limites de memória e tempo) em \textit{pipelines} de avaliação automática de código gerado por LLMs.

\subsection{Falhas Estruturais versus Estocásticas}

A comparação entre os dois experimentos permite classificar as falhas observadas em duas categorias.

\textbf{Falhas estruturais} persistem independentemente da temperatura, indicando uma limitação genuína do modelo para aquele \textit{prompt}. Para o \texttt{qwen2.5-coder}, a única falha estrutural identificada é P5 — os prompts verboso e vago consistentemente induzem bugs (tipo de retorno errado ou uso de \texttt{set()}), tanto com temperatura zero quanto na média das cinco runs, ainda que os bugs específicos gerados variem. Para o \texttt{deepseek-coder}, \texttt{P2\_vago} reprova de forma perfeitamente consistente em ambos os experimentos (0\% em temperatura zero e nas cinco execuções padrão); \texttt{P1\_verboso} é quase tão consistente — falha em temperatura zero e em quatro das cinco execuções padrão, com um único acerto isolado (20\,±\,45) —, e nenhuma variante de P2 atinge 100\% \textit{em média} (o máximo observado em qualquer execução isolada é 100\%, mas apenas esporadicamente, em \texttt{P2\_ingles} e \texttt{P2\_verboso}).

\textbf{Falhas estocásticas} aparecem apenas com temperatura padrão e desaparecem com temperatura zero — ou o contrário. São resultado da amostragem probabilística que eventualmente seleciona tokens de um caminho incorreto. Os exemplos mais claros estão no \texttt{qwen2.5-coder}: \texttt{P4\_ingles} (\texttt{str.count} em algumas runs, \textit{loop} correto com temperatura zero) e \texttt{P2\_com\_ruido} (normalização incompleta em algumas runs, correta com zero). No \texttt{deepseek-coder} a componente estocástica é difusa — a maioria das combinações apresenta desvio padrão elevado (Tabela~\ref{tab:multi_runs}), oscilando entre acerto e falha sem um caso isolado tão nítido.

Esses contrastes reforçam o argumento de Bommasani et al.~\cite{bommasani2023helm}: a avaliação por um único \textit{prompt} — ou por uma única execução — pode levar a conclusões opostas sobre a mesma capacidade, o que justifica a cobertura multidimensional (múltiplas variantes e múltiplas execuções) adotada neste trabalho.

Essa distinção tem implicação metodológica direta: \textbf{avaliar LLMs com temperatura padrão e execução única superestima falhas do modelo} (ao reportar como estrutural o que é estocástico) e \textbf{também pode superestimar capacidades} (ao considerar um acerto ocasional como comportamento confiável). Temperatura zero com execução única é o mínimo necessário para resultados reproduzíveis; temperatura padrão com múltiplas execuções é necessária para caracterizar o comportamento real de uso.

\section{Limitações, riscos e ameaças à validade}
% =============================================================================

\subsection{Número de amostras por combinação}

O experimento com temperatura padrão foi executado 5 vezes, permitindo calcular média e desvio padrão para cada combinação e superando a limitação de execução única presente nos trabalhos de referência. O experimento com temperatura zero, por ser determinístico, requer apenas uma execução. Para trabalhos futuros, recomenda-se aumentar o número de runs com temperatura padrão para pelo menos 10, a fim de obter estimativas de desvio padrão mais estáveis, especialmente para o \texttt{deepseek-coder}, cujos desvios próximos de 45–55pp indicam distribuição aproximadamente binomial (ou acerta ou não).

\subsection{Cobertura dos testes unitários}

As baterias de testes foram elaboradas manualmente pelo grupo. É possível que existam casos extremos não cobertos que permitiriam que implementações incorretas passassem em todos os testes. Uma abordagem mais robusta envolveria geração automatizada de casos de teste ou uso de ferramentas de \textit{fuzzing}.

\subsection{Escopo dos problemas}

Os cinco problemas selecionados são simples e algorítmicos. Os resultados obtidos não podem ser generalizados para tarefas mais complexas, como geração de código com múltiplas funções interdependentes, manipulação de estruturas de dados complexas ou integração com bibliotecas externas. A simplicidade dos problemas pode ter favorecido artificialmente o desempenho dos modelos.

\subsection{Variantes de prompt}

As variações de \textit{prompt} foram elaboradas pelo grupo sem protocolo formal de diversidade linguística. Embora este trabalho aplique seis tipos de variante simetricamente a todos os problemas, os prompts foram construídos manualmente. Trabalhos como PromptRobust \cite{zhu2023promptrobust} utilizam perturbações mais sistemáticas (substituição de caracteres, erros tipográficos, negações) que poderiam revelar fragilidades adicionais não cobertas pelos ensaios realizados.

\subsection{Ausência de acesso ao mecanismo interno}

Conforme estabelecido na situação-problema, a avaliação foi realizada exclusivamente a partir do comportamento observado. Não foi possível inspecionar pesos, mecanismos de atenção ou distribuições de probabilidade internas aos modelos. Isso limita a capacidade explicativa dos resultados — é possível identificar \textit{que} o \texttt{deepseek-coder} falha em determinados prompts, mas não \textit{por que} a partir dos experimentos realizados. A distinção entre falhas estruturais e estocásticas, realizada neste trabalho via comparação de temperaturas, oferece uma evidência comportamental, mas não substitui a análise interna dos modelos.

\subsection{Execução única por modelo}

Os experimentos foram realizados com uma única versão quantizada de cada modelo (versão \texttt{:1.5b} e \texttt{:1.3b}). Versões maiores dos mesmos modelos poderiam apresentar comportamento substancialmente diferente, e os resultados não devem ser interpretados como avaliação definitiva das famílias de modelos, mas apenas das versões específicas testadas.

    \item \textbf{Experimento 3 — 4 runs adicionais com temperatura padrão (240 combinações):} $\approx$ 30 minutos.
    \item \textbf{Total acumulado (360 chamadas à API local):} $\approx$ 46 minutos.
\end{itemize}
